# CANsub Intro

> Get started guide for the CANsub (CANsub.2 / CANsub.4) CAN-FD bus interface family. This documentation covers introduction, device configuration via the browser-based webCAN tool, protocol-specific guidance (OBD2, UDS, J1939, NMEA 2000, CANopen, CCP/XCP), streaming troubleshooting, and software tools (webCAN, Python via python-can-cansub, SavvyCAN, PlotJuggler, and custom REST/WebSocket apps). It is strongly recommended to also download the technical CANsub.2 Docs and CANsub.4 Docs and use these in combination with this introduction material. Those Docs are also available in AI/LLM-friendly text file versions.

# Streaming troubleshooting tips

The below steps can be useful when troubleshooting streaming issues with the CANsub.

> ##### Table of Contents
> 
> * [Check the PC connection](#check-the-pc-connection)
> * [Check the CAN bus connection](#check-the-can-bus-connection)
> * [Contact us](#contact-us)

---

## Check the PC connection

1. Verify that the LCD shows a device URL when you power it via USB<sup>[1](#usb)</sup>
2. Verify that webCAN opens on your PC when you enter the URL in your browser<sup>[2](#certificate)</sup>
3. For Ethernet usage, verify that the network runs a DHCP server<sup>[3](#mac)</sup>

*Once you can access webCAN you can move on to the next step.*

---

## Check the CAN bus connection

1. Verify that you are using an adapter cable with the right pin-out<sup>[4](#adapter)</sup>
2. Verify that the CAN bus is properly terminated<sup>[5](#termination)</sup>
3. Verify that the CAN bus is actively broadcasting CAN frames<sup>[6](#broadcast)</sup>
4. Verify that you are using the correct bit-rate
5. If the bus consists of a single node (other than the CANsub), disable ‘listen only’ mode<sup>[7](#ack)</sup>
6. If you are still not seeing CAN data, enable ‘error frames’ and try again<sup>[8](#errorframes)</sup>
7. In the webCAN ‘status bar’ inspect the RX/TX error counters<sup>[9](#errorcounters)</sup>

---

## Contact us

If you’re still unable to stream data, please contact us with:

1. Full details on your setup (CAN bus, bit-rate, termination, adapter cable, power supply)
2. CANsub variant (CANsub.2 or CANsub.4), firmware version and a picture of the LCD
3. A screenshot of webCAN (incl. the channel state + error counters)

---
* <a id='usb'>**[1]**</a> We recommend to test the simplest setup first, i.e. connect the CANsub via USB to your PC. If the device does not power on in this scenario, try with a different USB adapter cable
* <a id='certificate'>**[2]**</a> As per the ‘get started’ guide in the CANsub Docs you may see a warning when you try to access webCAN if you have not installed the TLS certificate. This is harmless and you can simply click “Proceed”. However, if your organization blocks this on your PC you will need to install the TLS certificate first (or ask IT to install it). See the step-by-step [certificate installation guide](https://canlogger.csselectronics.com/cansub-docs/cansub2/getting_started/certificate/index.html)
* <a id='mac'>**[3]**</a> If the network requires ‘white listing’ of devices you can get the device MAC address from the device label or display as per the Docs
* <a id='adapter'>**[4]**</a> Using an adapter cable with an incorrect pinout may damage the device/application and is not covered by the device warranty. If in doubt, contact us. See also our [DB25-4xDB9](https://www.csselectronics.com/products/db25-4x-db9-adapter-cable-can-bus) for the CANsub.4 and our [standard DB9 adapter cables](https://www.csselectronics.com/pages/can-bus-hardware-products#adapter-cables). Note that you must provide both CAN high, CAN low and ground for each CAN channel
* <a id='termination'>**[5]**</a> Already-active CAN buses (e.g. in cars, trucks, …) will be properly terminated (as they would otherwise not function properly). In contrast, test bench setups with e.g. an ECU, sensor-to-CAN module etc may require that you add [terminal resistors](https://www.csselectronics.com/products/terminal-resistor-can-bus)
* <a id='broadcast'>**[6]**</a> If you are e.g. connecting to an OBD2 connector in a car, there is often no CAN bus data broadcast (it is blocked by a gateway filter). In such a case, the webCAN bit-rate auto-detection does not work and you will need instead to follow our [OBD2 guide](../stream-data/obd2-data.md#ref-record-obd2-data) to request data
* <a id='ack'>**[7]**</a> If you e.g. aim to record data from a single CAN simulation device (like a sensor-to-CAN module), the communication requires that a CAN node acknowledges the frames being sent by the producer node. If the CANsub is configured in ‘listen only’ mode, the communication will fail in this case
* <a id='errorframes'>**[8]**</a> If you observe CAN errors in the Summary view, it indicates that something is wrong in the installation (e.g. the bit-rate, termination, …)
* <a id='errorcounters'>**[9]**</a> Non-zero error counters may indicate general problems on the bus. For example, if you’ve tried to connect to a CAN node with the wrong bit-rate, the node may have entered an error-passive state and might require power cycling to be reset

## Tips & tricks

Below are some general recommendations when building custom CANsub apps:

### Dynamic input fields

To ensure an app is reusable across different CAN applications and devices, it is **strongly recommended** to always include the below fields (see also [this Python demo](python/demos/cansub-plot.md#ref-python-demo-cansub-plot)):

* Device endpoint (e.g. `https://1f72d84a-usb.local`) - optionally with mDNS-based auto-detection
* CAN channel (e.g. dropdown with `CAN1`, `CAN2`)<sup>[1](#channels)</sup>
* Nominal bit-rate (e.g. `500000`)
* Data bit-rate (e.g. `1000000`)
* DBC file loader
* A ‘connect/disconnect’ button (never ‘auto connect on boot’ - the bit-rate may e.g. be wrong)

<br/>

### Python vs. Qt vs. web

When you aim to build a custom app, your first choice will be what platform to start from.

For most use cases, **Python** will be a good starting point as it lets you quickly create very simple apps - while also enabling advanced functionality due to the extensive add-on tools that work with python-can (see e.g. our [protocols](../stream-data/protocols.md#ref-record-protocols) section). If you are just building an ad hoc tool for use by yourself or your own team, this is often the best starting point.

The downside of Python is sharing: End users either need Python installed, or you bundle the app into an `*.exe` (e.g. via PyInstaller), which tends to be large, per-OS and prone to antivirus flags<sup>[2](#flask)</sup>. For polished desktop apps you distribute to others, a **Qt plugin** app (C++) instead compiles to a lean native `*.exe` - at the cost of more complex development and CAN libraries/packages that are not as rich as in Python.

**Web apps** (like webCAN) offer the unique advantage that they work on any operating system (Windows, Linux, MacOS) and any device (PC, tablet, phone) out-the-box. Configuration and data processing is done purely in the browser, meaning users do not have to install any software to use the app<sup>[3](#tls)</sup>. The CANsub is uniquely suited for web apps as it will be exposed as an identical network interface via both USB/Ethernet - with a web-native REST/WebSocket API. A web app can be compiled e.g. to a simple HTML file that can be used offline, or hosted on a local network for sharing in a team. In other words, web apps offer unique convenience for end users and offer many useful open source libraries like Python - however, for CAN-specific protocol implementations you will need to build more things from scratch.

---
* <a id='channels'>**[1]**</a> The number of available CAN channels can be requested from a CANsub device via the REST API, so the dropdown can be populated dynamically per device
* <a id='flask'>**[2]**</a> A useful tip is to build Python apps with [Flask](https://flask.palletsprojects.com/) (or similar): A single host PC/Raspberry Pi on the network runs the app and exposes it via a URL, so other teams can use it with CANsubs on the network from their browser - without installing anything on their own machines (see e.g. [this Python demo](python/demos/cansub-plot.md#ref-python-demo-cansub-plot))
* <a id='tls'>**[3]**</a> The CANsub API is TLS-only, so the browser must trust the device certificate. Users either accept a one-time browser security warning or install the CANsub root certificate (see the [certificate installation guide](https://canlogger.csselectronics.com/cansub-docs/cansub2/getting_started/certificate/index.html)). A web app served by the device (like webCAN) is same-origin, whereas a separately hosted/offline web app connects cross-origin

# CANedge

The [CANedge](https://www.csselectronics.com/pages/can-bus-hardware-products) is our series of CAN bus data loggers. In this section we explain how you can use your CANedge and CANsub devices together.

> ##### Table of Contents
> 
> * [Load CANedge log files in webCAN](#load-canedge-log-files-in-webcan)
> * [Convert CANedge transmit lists](#convert-canedge-transmit-lists)
>   * [Example 1: Convert CANedge transmit list to CANsub webCAN config](#example-1-convert-canedge-transmit-list-to-cansub-webcan-config)
>   * [Example 2: Convert CANedge transmit list to Python sequence](#example-2-convert-canedge-transmit-list-to-python-sequence)

---

## Load CANedge log files in webCAN

The CANedge logs data to an SD card in the binary MF4 format. Using the free [MF4 converters](https://www.csselectronics.com/screen/page/mdf4-converters-mf4-asc-csv), you can convert these log files to other formats - including CSV via the `mdf2csv` converter. The resulting CSV output can be loaded directly in webCAN for e.g. analysis and plotting of your recorded data, letting you re-use the same webCAN workflow for both live CANsub streams, exported webCAN CSV traces and historical CANedge log files.

If you use the CANsub with Python, you can also natively load the raw CSV files. Further you can optionally install [asammdf](https://github.com/danielhrisca/asammdf) to directly load the CANedge MF4 logs e.g. for replay via the CANsub.

---

## Convert CANedge transmit lists

Both the CANedge and CANsub can transmit CAN frames, but they use different schemas. To re-use CANedge transmit lists with your CANsub, we provide a small helper module:

[`canedge_utils.py`](/_static/files/stream/canedge/canedge_utils.py)

It offers two conversion paths - one targeting a CANsub webCAN config (for on-device deployment) and one targeting a python-can transmit schedule (for host-side streaming).

<br/>

### Example 1: Convert CANedge transmit list to CANsub webCAN config

You can convert a CANedge Configuration File into a webCAN-compatible CANsub config to e.g. deploy an existing transmit sequence onto the device. The `convert_transmit_config` function reads the CANedge transmit lists (CAN1 / CAN2) and outputs a partial CANsub configuration file containing only the transmit section, which you can then import in webCAN.

```python
import json
from canedge_utils import convert_transmit_config

# load CANedge configuration file
with open("config-01.09.json") as f:
    cansub_config = convert_transmit_config(json.load(f))

# output partial CANsub configuration file
with open("config-02.00.json", "w") as f:
    json.dump(cansub_config, f, indent=2)
```

#### NOTE
webCAN cannot express variable interframe spacing - a CANedge channel with uneven spacing between frames is not convertible to a single webCAN sequence<sup>[1](#spacing)</sup>

<br/>

### Example 2: Convert CANedge transmit list to Python sequence

You can also load a CANedge Configuration File directly in [Python](../software-tools/python.md#ref-software-python) for use in e.g. a custom transmit list streamed by the CANsub. The `load_transmit_schedule` function parses the CANedge transmit list into a `PERIOD` and `SCHEDULE` that you can stream client-side:

```python
from time import sleep, monotonic
import can
from canedge_utils import load_transmit_schedule

CHANNEL = "1f72d84a-usb.local@1"
CONFIG = "config-01.09.json"

# PERIOD and SCHEDULE come from the CANedge config transmit list, not hardcoded.
PERIOD, SCHEDULE = load_transmit_schedule(CONFIG)

with can.Bus(interface="cansub", channel=CHANNEL,
             bitrate=500_000, data_bitrate=1_000_000) as bus:

    # Stream received frames; transmit is host-scheduled below (nothing deployed to the device).
    with can.Notifier([bus], listeners=[can.Printer()]):
        try:
            cycle = monotonic()
            while True:
                for offset, msg in SCHEDULE:
                    sleep(max(0.0, cycle + offset - monotonic()))
                    bus.send(msg)
                cycle += PERIOD
                sleep(max(0.0, cycle - monotonic()))
        except KeyboardInterrupt:
            print("\nstopped")
```

This variant is useful when the CANedge transmit list uses mixed interframe spacing - i.e. when it cannot be deployed as a single on-device webCAN sequence. Here the Python script schedules each frame at its own offset, so any spacing pattern is supported.

---
* <a id='spacing'>**[1]**</a> The on-device webCAN transmit sequence uses a single period and a single (constant) spacing between frames. If your CANedge channel mixes different spacings, use the Python sequence variant below instead

# Stream CANopen data

CANopen is a protocol used in industrial machinery, robotics, and some automotive applications.

In this section we provide tips on how to stream CANopen data with your CANsub.

## CANopen DBC vs EDS files

To DBC decode CANopen data in e.g. webCAN you need the application-specific DBC file. CANopen networks normally ship with an EDS (Electronic Data Sheet) rather than a DBC. You can convert your EDS to a DBC as outlined in our [EDS-to-DBC guide](https://www.csselectronics.com/pages/canopen-eds-file-electronic-data-sheet#convert-eds-to-dbc) for use in e.g. webCAN. If you use the CANsub in Python, you can natively use EDS files.

---

## Stream CANopen data (webCAN)

To stream CANopen data from e.g. an industrial machine, you need to find a suitable way to connect the CANsub. Below are the most common methods:

1. [M12-DB9](https://www.csselectronics.com/products/m12-db9-cable-5-pin): This lets you stream CANopen data from applications using an M12 connector
2. [DB9](https://www.csselectronics.com/pages/can-bus-hardware-products#adapter-cables): A standard DB9 adapter cable is often suitable for CANopen applications
3. [Contactless CH1](https://www.csselectronics.com/products/contactless-can-bus-reader-adapter): This lets you stream CANopen data directly from the CAN high/low harness

Once you have identified which adapter cable to use you can follow below steps:

1. Connect the CANsub to your CANopen application via the adapter cable
2. In webCAN, load the CANopen DBC file and assign it to CAN1
3. Use the bit-rate ‘auto-detect’ mode and enable ‘Listen Only’, then click connect
4. Verify that the incoming raw data appears in the Trace/Summary view
5. Check if you can DBC decode the data and e.g. plot it

CANopen Process Data Objects (PDOs) are broadcast and require no transmit configuration. To actively request data via Service Data Objects (SDOs), you can configure a transmit sequence in webCAN for the SDO upload request.

![canopen-dbc-decode-webcan-pdo-sdo](https://www.csselectronics.com/cdn/shop/files/canopen-dbc-decode-webcan-pdo-sdo-c.png)

---

## Stream CANopen data (Python)

You can also stream CANopen data via the CANsub [Python API](../software-tools/python.md#ref-software-python).

Below is a simple example showcasing DBC decoding of raw CANopen PDO/SDO data:

```python
import can, cantools

CHANNEL = "1f72d84a-usb.local@1"

db = cantools.database.load_file("canopen-node.dbc", strict=False)
pdo = db.get_message_by_name("TPDO1_Com_State")   # PDO example (COB-ID 0x182)
sdo = db.get_message_by_name("SDO_transmit")      # SDO upload response example (COB-ID 0x582)

with can.Bus(interface="cansub", channel=CHANNEL, bitrate=250_000, data_bitrate=1_000_000) as bus:
    for frame in bus:
        if frame.arbitration_id == pdo.frame_id:
            value = pdo.decode(frame.data)["Application_State"]
            print(f"{frame.timestamp:.3f}  TPDO1_Com_State.Application_State = {value}")
        elif frame.arbitration_id == sdo.frame_id:
            decoded = sdo.decode(frame.data, allow_truncated=True, decode_choices=False)
            if "Number_of_entries1" in decoded:
                print(f"{frame.timestamp:.3f}  SDO_transmit.Number_of_entries1 = {decoded['Number_of_entries1']}")
```

<br/>

### Advanced: CANopen EDS decoding

You can also use the [canopen](https://github.com/canopen-python/canopen) package to stream CANopen data with direct support for EDS files:

```python
from time import sleep
import canopen

CHANNEL = "1f72d84a-usb.local@1"

EDS = "canopen-node.eds"
NODE_ID = 2

# open the bus and load node 2 from its EDS
network = canopen.Network()
network.connect(interface="cansub", channel=CHANNEL, bitrate=250_000, data_bitrate=1_000_000)
with open(EDS, encoding="latin-1") as f:
    node = network.add_node(NODE_ID, f)

# build the TPDO maps from the EDS and subscribe their COB-IDs for decoding
node.tpdo.read(from_od=True)

# print each mapped signal's physical value (fires on every received TPDO)
def on_pdo(pdomap):
    for var in pdomap:
        print(f"{var.name} = {var.phys}")

for n in node.tpdo:
    node.tpdo[n].add_callback(on_pdo)

try:
    while True:
        sleep(1)
except KeyboardInterrupt:
    pass
finally:
    network.disconnect()
```

<br/>

### Advanced: CANopen master / SDO client

The CANsub can also operate as a CANopen master / SDO client on the network:

```python
from time import sleep
import canopen

CHANNEL = "1f72d84a-usb.local@1"

EDS = "canopen-node.eds"
NODE_ID = 2

# the CANsub is the master / SDO client (it has no node id of its own here)
network = canopen.Network()
network.connect(interface="cansub", channel=CHANNEL, bitrate=250_000, data_bitrate=1_000_000)

# register a handle to the remote device we want to read (node id 2), using its EDS
with open(EDS, encoding="latin-1") as f:
    node = network.add_node(NODE_ID, f)

# each .raw access sends an SDO upload request and returns the response, decoded via the EDS
try:
    while True:
        print(f"Device Type = 0x{node.sdo[0x1000].raw:08X}")
        print(f"Manufacturer device name = {node.sdo['Manufacturer device name'].raw}")
        sleep(1.0)
except KeyboardInterrupt:
    pass
finally:
    network.disconnect()
```

### Advanced: Multi-frame messages (J1939 TP)

### Advanced: Request J1939 data

In some cases, you may wish to stream on-request J1939 data. To do so, you need to transmit the relevant J1939 request frame into the CAN bus - see also our [J1939 simple intro](https://www.csselectronics.com/screen/page/simple-intro-j1939-explained/language/en).

An example is PGN `65253` (`FEE5`) which contains the SPN Engine Hours. To request this message you can leverage webCAN’s DBC encoder dialog:

1. In the ‘Transmit Sequences’ panel click ‘Encode from DBC’ on the default entry
2. Search for the `RQST` PGN and set the signal value to the decimal PGN, `65253`
3. Note that the transmit CAN ID is `18EAFFFE`, reflecting the J1939 request PGN `59904` (`EA00`)
4. Note that we use a ‘global request’ (`0xFF`) and an arbitrary source address (`0xFE`)

![CANedge J1939 on-request transmit frame](https://www.csselectronics.com/cdn/shop/files/j1939-request-message-webcan-stream-engine-hours-c.png)

#### NOTE
A vehicle might not respond to your request - and caution should be taken when transmitting<sup>[4](#fn-transmit)</sup>

## Stream J1939 data (Python)

You can also stream J1939 data via the CANsub [Python API](../software-tools/python.md#ref-software-python) incl. PGN-based DBC decoding:

import can, cantools
from cantools.j1939 import pgn_from_frame_id

CHANNEL = "1f72d84a-usb.local@1"

### Advanced: Multi-frame messages (J1939 TP)

You can use the [can-j1939](https://github.com/juergenH87/python-can-j1939) package for J1939 TP decoding<sup>[5](#tpwarning)</sup>:

```python
import time
import cantools, j1939
from cantools.j1939 import pgn_from_frame_id

CHANNEL = "1f72d84a-usb.local@1"
SIGNALS = ("EngineSpeed", "EnginePercentTorqueAtPoint2")

db = cantools.database.load_file("j1939.dbc", strict=False)
by_pgn = {pgn_from_frame_id(m.frame_id): m for m in db.messages if any(s.name in SIGNALS for s in m.signals)}

def on_message(_priority, pgn, _sa, timestamp, data):
    msg = by_pgn.get(pgn)
    if msg:
        decoded = msg.decode(bytes(data), allow_truncated=True)
        for spn in SIGNALS:
            if spn in decoded:
                print(f"{timestamp:.3f}  {msg.name}.{spn} = {decoded[spn]}")

ecu = j1939.ElectronicControlUnit()
ecu.connect(interface="cansub", channel=CHANNEL, bitrate=250_000, data_bitrate=1_000_000)
ecu.subscribe(on_message)

try:
    while True:
        time.sleep(1)
except KeyboardInterrupt:
    pass
finally:
    ecu.disconnect()
```

#### NOTE
The above script can also be used for variable-length messages like the J1939-73 DM1

* <a id='pgndecoding'>**[1]**</a> PGN decoding is automatically enabled if your J1939 DBC file sets the `ProtocolType` attribute to `J1939`. This also means that any ‘PGN conflicts’ at the bus level across your assigned DBC files will be flagged as invalid by webCAN in this case
* <a id='dbcsyntax'>**[2]**</a> If your J1939 DBC sets the `ProtocolType` attribute to `J1939`, the J1939 TP rules will automatically be applied when you assign the DBC to a CAN channel
* <a id='variablelength'>**[3]**</a> In your J1939 DBC, add a `VariableLength` integer attribute and set the value to `1` for variable-length messages like the DM1
* <a id='fn-transmit'>**[4]**</a> Be careful if transmitting data onto a vehicle while the ignition is off (as this may ‘wake up’ the ECUs and cause battery drain)
* <a id='tpwarning'>**[5]**</a> The `ecu.subscribe` takes an optional `device_address` field, by default set to `None`. If set to a specific source address (e.g. `0xF9`) the CANsub may attempt to respond to peer-to-peer TP communication directed at that source address, which should not be done in most use cases

### Advanced: Multi-frame messages (Fast Packets)

Unlike J1939 TP and ISO-TP, there is not a great plug & play python-can library for NMEA 2000 Fast Packets. Below is a very minimal starting point showcasing how this can be implemented:

```python
import can, cantools
from cantools.j1939 import pgn_from_frame_id

CHANNEL = "1f72d84a-usb.local@1"
SIGNALS = ("EngineOilPressure", "EngineTemp", "AlternatorPotential", "FuelRate")

db = cantools.database.load_file("NMEA-2000-3.0-v5.3.dbc", strict=False)
by_pgn = {pgn_from_frame_id(m.frame_id): m for m in db.messages if any(s.name in SIGNALS for s in m.signals)}

def fast_packet(sessions, key, data):
    sequence, counter = data[0] >> 5, data[0] & 0x1F
    if counter == 0:
        sessions[key] = {"sequence": sequence, "total": data[1], "buf": bytearray(data[2:8]), "next": 1}
    else:
        s = sessions.get(key)
        if not s or s["sequence"] != sequence or s["next"] != counter:
            sessions.pop(key, None)
            return None
        s["buf"] += data[1:8]
        s["next"] += 1
        if len(s["buf"]) >= s["total"]:
            sessions.pop(key)
            return bytes(s["buf"][: s["total"]])
    return None
# ----------------------------------------------------------------------------------------

sessions = {}
with can.Bus(interface="cansub", channel=CHANNEL, bitrate=250_000, data_bitrate=1_000_000) as bus:
    for frame in bus:
        pgn = pgn_from_frame_id(frame.arbitration_id)
        msg = by_pgn.get(pgn)
        if not msg:
            continue
        payload = fast_packet(sessions, (pgn, frame.arbitration_id & 0xFF), frame.data)
        if payload:
            decoded = msg.decode(payload, allow_truncated=True, decode_choices=False)
            for sig in SIGNALS:
                if sig in decoded:
                    print(f"{frame.timestamp:.3f}  {msg.name}.{sig} = {decoded[sig]}")
```

---
* <a id='pgndecoding'>**[1]**</a> PGN decoding is automatically enabled if your DBC file sets the `ProtocolType` attribute to `J1939`. This also means that any ‘PGN conflicts’ at the bus level across your assigned DBC files will be flagged as invalid by webCAN in this case
* <a id='dbcsyntax'>**[2]**</a> If you use our NMEA 2000 DBC, the TP syntax is already setup correctly. If you use your own DBC, add the string attribute `TransportProtocolType` and assign this to each NMEA 2000 Fast Packet ID with the value `NMEATP` to make webCAN apply the NMEA 2000 TP rules automatically when you assign the DBC to a CAN channel

## Stream OBD2 data (Python)

You can achieve the same result via the CANsub [Python API](../software-tools/python.md#ref-software-python). Below is a minimal script to request and stream OBD2 data in real-time with edge-based precision:

```python
from time import sleep
import can

CHANNEL = "1f72d84a-usb.local@1"

PID_REQUESTS = [
    [0x02, 0x01, 0x0C, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF],  # Engine RPM
    [0x02, 0x01, 0x0D, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF],  # Vehicle speed
    [0x02, 0x01, 0x04, 0xFF, 0xFF, 0xFF, 0xFF, 0xFF],  # Calc. engine load
]

msgs = [
    can.Message(is_extended_id=False, arbitration_id=0x7DF, data=data)
    for data in PID_REQUESTS
]

with can.Bus(interface="cansub", channel=CHANNEL,
             bitrate=500_000, data_bitrate=1_000_000) as bus:

    with can.Notifier([bus], listeners=[can.Printer()]):
        bus.send_periodic(msgs, period=0.3, duration=None)

        try:
            while True:
                sleep(1)
        except KeyboardInterrupt:
            print("\nstopped")
```

### Mixed spacing requests

The below example shows how you can create more custom transmit sequences and manage these purely client-side (without deploying them on the CANsub):

```python
from time import sleep, monotonic
import can

CHANNEL = "1f72d84a-usb.local@1"
PERIOD = 5.0  # seconds; shared cycle length for all messages

SCHEDULE = [
    (0.000, can.Message(is_extended_id=False, arbitration_id=0x7E4, data=[0x03, 0x22, 0x01, 0x01, 0xAA, 0xAA, 0xAA, 0xAA])),  # SF_BMS
    (0.030, can.Message(is_extended_id=False, arbitration_id=0x7E4, data=[0x30, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00])),  # FC_BMS
    (0.500, can.Message(is_extended_id=False, arbitration_id=0x7B3, data=[0x03, 0x22, 0x01, 0x00, 0xAA, 0xAA, 0xAA, 0xAA])),  # SF_Temperatures
    (0.530, can.Message(is_extended_id=False, arbitration_id=0x7B3, data=[0x30, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00])),  # FC_Temperatures
]

with can.Bus(interface="cansub", channel=CHANNEL,
             bitrate=500_000, data_bitrate=1_000_000) as bus:

    with can.Notifier([bus], listeners=[can.Printer()]):
        try:
            cycle = monotonic()
            while True:
                for offset, msg in SCHEDULE:
                    sleep(max(0.0, cycle + offset - monotonic()))
                    bus.send(msg)
                cycle += PERIOD
                sleep(max(0.0, cycle - monotonic()))
        except KeyboardInterrupt:
            print("\nstopped")
```

#### NOTE
You can also load the `SCHEDULE` from a CANedge Configuration File (e.g. from the EV data pack) via the `canedge_utils.py` outlined in the [CANedge](canedge.md#ref-record-canedge) section

<br/>

### Advanced: Multi-frame UDS data (ISO-TP)

You can also use the [can-isotp](https://github.com/pylessard/python-can-isotp) package to handle ISO-TP for your requests and for multi-frame assembly, enabling subsequent DBC decoding:

```python
from time import sleep
import isotp, can, cantools

CHANNEL = "1f72d84a-usb.local@1"

db = cantools.database.load_file("kia-v2.4.dbc", strict=False)
battery = db.get_message_by_name("Battery")

with can.Bus(interface="cansub", channel=CHANNEL, bitrate=500_000, data_bitrate=1_000_000) as bus:
    addr = isotp.Address(isotp.AddressingMode.Normal_11bits, rxid=0x7EC, txid=0x7E4)
    stack = isotp.CanStack(bus, address=addr)  # isotp sends the request and the Flow Control
    stack.start()
    try:
        while True:
            stack.send(b"\x22\x01\x01")                     # UDS ReadDataByIdentifier, DID 0x0101
            payload = stack.recv(block=True, timeout=0.5)   # reassembled multi-frame response
            if payload:
                decoded = battery.decode(bytes(payload), allow_truncated=True)
                for name, value in decoded.items():
                    if "StateOfCharge" in name:
                        print(f"{name} = {value} %")
            sleep(1.0)
    except KeyboardInterrupt:
        pass
    finally:
        stack.stop()
```

---
* <a id='dbcsyntax'>**[1]**</a> In your DBC add a `TransportProtocolType` string attribute and set IDs as `ISOTP` using e.g. our [DBC editor](https://www.csselectronics.com/pages/dbc-editor-can-bus-database)

## Stream XCP data (Python)

The [pyxcp](https://github.com/christoph2/pyxcp) package enables you to set up advanced XCP communication via the CANsub. In this section we provide some guidance on how to get started.

### Reusable connection helper

The examples below share the same initialization code for using the CANsub with pyxcp. The subsequent examples assume this is saved as `cansub_xcp.py` next to the other scripts.

```python
import logging
import can
from pyxcp.transport.can import CanInterfaceBase
from pyxcp.config import create_application_from_config
from pyxcp.master import Master

# expose the cansub bus to pyxcp's transport layer
class CanSub(CanInterfaceBase):
    def __init__(self, channel, bitrate, data_bitrate):
        self.bus = can.Bus(interface="cansub", channel=channel, bitrate=bitrate, data_bitrate=data_bitrate)
    def init(self): pass
    def set_filters(self, filters=None): pass
    def recv(self, timeout=None): return self.bus.recv(timeout)
    def send(self, msg): self.bus.send(msg)
    @property
    def filters(self): return None
    @property
    def state(self): return can.BusState.ACTIVE
    def shutdown(self): self.bus.shutdown()

# build a pyxcp master bound to a cansub channel; returns (master, interface)
def make_master(channel, can_id_master, can_id_slave, bitrate, data_bitrate, fd=False, policy=None):
    iface = CanSub(channel, bitrate, data_bitrate)
    app = create_application_from_config(
        {"Transport": {"CAN": {"can_id_master": can_id_master, "can_id_slave": can_id_slave, "fd": fd}}},
        log_level=logging.WARNING,
    )
    app.transport.layer = "CAN"
    return Master("CAN", config=app, policy=policy, transport_layer_interface=iface), iface
```

<br/>

### Connect and read ECU information

Here the CANsub connects to the ECU and reads the CONNECT capabilities and GET_STATUS:

from cansub_xcp import make_master

CHANNEL = "1f72d84a-usb.local@1"
CAN_ID_MASTER, CAN_ID_SLAVE = 0x111, 0x222
BITRATE, DATA_BITRATE = 500_000, 1_000_000

finally:

### Seed & key authentication

Some ECUs protect resources (e.g. DAQ) behind a seed & key exchange. The CANsub can request a seed, derive the key and unlock the ECU to e.g. enable DAQ initialization:

```python
from cansub_xcp import make_master

CHANNEL = "1f72d84a-usb.local@1"
CAN_ID_MASTER, CAN_ID_SLAVE = 0x111, 0x222
BITRATE, DATA_BITRATE = 500_000, 1_000_000
RESOURCE_DAQ = 0x04  # the resource to unlock

# demo seed -> key transform (real ECUs use a secret DLL); must match the slave
def seed_to_key(seed):
    return bytes((b ^ 0x5A) for b in seed)

master, iface = make_master(CHANNEL, CAN_ID_MASTER, CAN_ID_SLAVE, BITRATE, DATA_BITRATE)
try:
    with master:
        master.connect()
        seed = bytes(master.getSeed(0, RESOURCE_DAQ).seed)
        key = seed_to_key(seed)
        print(f"seed={seed.hex()} -> key={key.hex()}")
        result = master.unlock(len(key), key)
        print(f"unlocked; still-protected: daq={result.daq} calpag={result.calpag} pgm={result.pgm}")
        master.disconnect()
finally:
    iface.shutdown()
```

<br/>

### Initialize a dynamic DAQ measurement (no A2L)

To stream XCP data you can set up a DAQ list. Here the measurement details are given directly (no A2L). After this, the CANsub initializes the DAQ sequence and decodes the incoming DTOs:

```python
import time
from pyxcp.daq_stim import DaqList, DaqOnlinePolicy
from cansub_xcp import make_master

CHANNEL = "1f72d84a-usb.local@1"
CAN_ID_MASTER, CAN_ID_SLAVE = 0x111, 0x222
BITRATE, DATA_BITRATE = 500_000, 1_000_000

# measurements of interest specified via raw addresses (no A2L) as (name, address, ext, datatype)
MEASUREMENTS = [("SpeedSignal", 0xFACD01A2, 0, "U32"), ("SignalQuality", 0xFACD41C1, 0, "U8")]

# print each decoded DAQ sample as it arrives
class PrintPolicy(DaqOnlinePolicy):
    def on_daq_list(self, daq_list, ts0, ts1, payload):
        print(f"DAQ{daq_list}: {payload}")

daq_list = DaqList(name="engine", event_num=0, stim=False, enable_timestamps=False, measurements=MEASUREMENTS)
policy = PrintPolicy([daq_list])

master, iface = make_master(CHANNEL, CAN_ID_MASTER, CAN_ID_SLAVE, BITRATE, DATA_BITRATE, policy=policy)
try:
    with master:
        master.connect()
        policy.setup(write_multiple=False)
        policy.start()
        time.sleep(5)
        policy.stop()
        master.disconnect()
finally:
    iface.shutdown()
```

<br/>

### Initialize a DAQ measurement from an A2L file (CAN FD)

In most cases you will prefer to use an [A2L file](https://www.csselectronics.com/pages/a2l-file-asap2-intro-xcp-on-can-bus), which can be loaded via the [pya2ldb](https://github.com/christoph2/pyA2L) package. This example also uses CAN FD and `WRITE_DAQ_MULTIPLE`:

```python
import time
from pya2l import DB
import pya2l.model as model
from pyxcp.daq_stim import DaqList, DaqOnlinePolicy
from cansub_xcp import make_master

CHANNEL = "1f72d84a-usb.local@1"
CAN_ID_MASTER, CAN_ID_SLAVE = 0x111, 0x222
BITRATE, DATA_BITRATE = 500_000, 1_000_000
A2L = "ECU-sample-file-xcp.a2l"

# measurements of interest (resolved against the A2L below)
MEASUREMENTS = ["SpeedSignal", "AccelerationSignal", "VelocitySignal", "TemperatureSignal"]

# map A2L datatype names to pyxcp datatype strings
A2L_TO_XCP = {"UBYTE": "U8", "SBYTE": "I8", "UWORD": "U16", "SWORD": "I16",
              "ULONG": "U32", "SLONG": "I32", "FLOAT32_IEEE": "F32", "FLOAT64_IEEE": "F64"}

# print each decoded sample as name->value
class PrintPolicy(DaqOnlinePolicy):
    def on_daq_list(self, daq_list, ts0, ts1, payload):
        print(dict(zip(MEASUREMENTS, payload)))

# resolve each measurement name to (name, address, ext, datatype) via the A2L
session = DB().import_a2l(A2L, in_memory=True)
daq_measurements = []
for name in MEASUREMENTS:
    m = session.query(model.Measurement).filter(model.Measurement.name == name).first()
    daq_measurements.append((name, m.ecu_address.address, 0, A2L_TO_XCP[m.datatype]))

daq_list = DaqList(name="from_a2l", event_num=0, stim=False, enable_timestamps=False, measurements=daq_measurements)
policy = PrintPolicy([daq_list])

master, iface = make_master(CHANNEL, CAN_ID_MASTER, CAN_ID_SLAVE, BITRATE, DATA_BITRATE, fd=True, policy=policy)
try:
    with master:
        master.connect()
        policy.setup(write_multiple=True)
        policy.start()
        time.sleep(5)
        policy.stop()
        master.disconnect()
finally:
    iface.shutdown()
```

---
* <a id='limits'>**[1]**</a> The CANedge supports up to 224 CAN frames to be transmitted. The CANsub supports up to 64 sequences of up to 32 frames each - i.e. up to 2048 CAN frames in total. However, while the interframe spacing is extremely precise within a sequence, cross-sequence timing is subject to jitter related to e.g. the timing at which each sequence is started by the user. Future firmware updates may address these limitations, but until then more complex dynamic DAQ sequences should be initialized using e.g. the pyxcp tools

# Configure

webCAN is the most common way to configure your CANsub device settings. However, under-the-hood webCAN just leverages the CANsub’s open REST API (see the Docs for details). This is described by an [OpenAPI](https://www.openapis.org/) file, letting you use any programming language to configure the device.

---

## Configuration via python-can

For most Python use cases, our python-can integration abstracts away the REST-based configuration through the python-can interface. For example, the below uses REST under-the-hood to set the bit-rate and to deploy a transmit sequence on the CANsub:

```python
import can
from time import sleep

CHANNEL = "1f72d84a-usb.local@1"

msgs = [
    can.Message(is_extended_id=False, arbitration_id=0x123, data=[0x01, 0x02, 0x03, 0x04]),
]

with can.Bus(interface="cansub", channel=CHANNEL, bitrate=250_000, data_bitrate=1_000_000) as bus:
    # period: time between frames - duration: total transmit time (deployed as a hardware sequence)
    task = bus.send_periodic(msgs, period=0.1, duration=5.0)
    sleep(6)
```

---

## Configuration via REST

A CANsub config change can also be applied via the REST API via the `requests` module:

```python
import requests
from python_can_cansub import CANSUB_ROOT_CERT

HOST = "1f72d84a-usb.local"
CHANNEL = 1

sequence = {"limit": 50, "period_ms": 100, "spacing_ms": 1, "frames": [{"is_extended": False, "is_fd": False, "is_rtr_brs": False, "id": 0x123, "data": "01020304"}]}

r = requests.post(f"https://{HOST}/api/can/{CHANNEL}/transmit", json=sequence, verify=CANSUB_ROOT_CERT)
r.raise_for_status()
print(f"Transmit sequence configured: {r.status_code}")
```

## Configure device

<br/>

#### NOTE
For detailed guidance on the device configuration, see the [CANsub Docs](https://www.csselectronics.com/pages/can-bus-hardware-software-docs)

### PHY settings

You can set the **bit-rate** per CAN channel either manually or use auto-detection<sup>[1](#autodetect)</sup>. You can also enable **listen only** mode, **auto reset** (to auto-recover from bus-off) and **error frames**.

### Device filters

You can set **device CAN ID filters**. These apply on the device and are useful in reducing network load on high-throughput buses. The default filters allow all CAN IDs to pass through.

<a id="ref-webcan-transmit"></a>

### Transmit sequences

<br/>

You can define multi-frame transmit sequences per channel with custom period, inter-frame spacing and an optional count limit. The **DBC encoder** dialog lets you type physical signal values from assigned [DBC files](#ref-webcan-decode) (incl. PGN / multiplexing support). Sequences are deployed on the device and run with edge-based scheduling precision - independent of the host PC, network latency etc.

### Export/import

Device settings made in webCAN can be **exported** as a JSON configuration file<sup>[3](#export)</sup> and **re-imported** later on the same (or a different) CANsub. webCAN also accepts ‘partial’ JSON files<sup>[4](#partial)</sup>.

---

<a id="ref-webcan-transport-protocols"></a>

## Transport protocols

<br/>

webCAN can automatically reassemble multi-frame transport-protocol payloads for **ISO-TP**, **J1939 TP** and **NMEA 2000 Fast Packets**. TP rules can be set manually or by assigning a [DBC file](#ref-webcan-decode) with the relevant syntax. See the ‘help’ icon in webCAN or our [protocols](../../stream-data/protocols.md#ref-record-protocols) section for details.

The assembled frames appear along the raw frames in the [trace and summary](summary-trace-chart.md#ref-webcan-summary-trace-chart) views and can be DBC decoded like regular frames.

---

<a id="ref-webcan-decode"></a>

## Decode

<br/>

In webCAN you can load [DBC files](https://www.csselectronics.com/pages/can-dbc-file-database-intro) and assign them to your CAN channels. This lets you decode raw CAN frames into physical values like degC, km/h, RPM etc.

You can assign multiple DBC files per CAN bus, with full support for PGN decoding, multiplexing and more. Loaded DBC files and their CAN channel assignments are cached in the browser’s local storage and persist across page reloads.

Once DBC files are assigned, you can [display/visualize](summary-trace-chart.md#ref-webcan-summary-trace-chart) the decoded signals.

#### NOTE
webCAN will reject invalid DBC files - our [DBC editor](https://www.csselectronics.com/pages/dbc-editor-can-bus-database) can help troubleshoot these<sup>[5](#duplication)</sup>

---
* <a id='autodetect'>**[1]**</a> Auto-detection only works if your CAN bus actively broadcasts CAN frames
* <a id='cookies'>**[2]**</a> The software settings are linked to the URL/IP you use to access your CANsub and can be cleared as normal cookies
* <a id='export'>**[3]**</a> The exported JSON configuration file from webCAN format mirrors the CANsub REST endpoints and payloads. This makes it easy to re-use in other tools like Python
* <a id='partial'>**[4]**</a> The partial JSON files must replicate the correct JSON tree structure, but can e.g. skip subsections like specific CAN channels/settings
* <a id='duplication'>**[5]**</a> If you assign multiple DBC files to a single CAN channel, webCAN rejects your assignment if the resulting set of DBC files have conflicting CAN IDs and/or PGNs

# Status bar

![webCAN status bar](https://www.csselectronics.com/cdn/shop/files/webcan-status-bar-data-c.png)

The status bar at the bottom of webCAN surfaces real-time information from your CANsub per CAN channel, including the CAN controller state, frame count, frames/second, busload (%) and receive error counters / transmit error counters.

Unlike the summary, trace and chart views - which are computed in your browser from the received frame stream - the status bar values are polled directly from the device at 1 Hz. They therefore reflect the CANsub’s own state in real time.

## Summary

<br/>

The **summary** view shows one row per unique ID / signal - giving you a quick overview of the bus. It works on raw frames or, when [DBC files](configuration.md#ref-webcan-decode) are loaded, on decoded signals. In decoded mode you can display either the MIN, MAX, AVG or LAST physical value per signal<sup>[1](#summarystats)</sup>.

The summary view also supports a ‘fade’ mode that hides stale data bytes, e.g. useful in reverse engineering. Further, the summary/trace views also let you optionally display meta info such as the PGN equivalent of CAN IDs or transport protocol related information.

---

<a id="ref-webcan-trace"></a>

## Trace

<br/>

The **trace** view shows every received/transmitted frame across channels, sorted by their device-generated timestamps (1 µs resolution). When [DBC files](configuration.md#ref-webcan-decode) are loaded you can display the decoded signal values related to each frame.

webCAN’s cyclic buffer carries up to 2M frames with automatic first-in-first-out eviction, so you can scroll back through history while new frames keep arriving.

---

<a id="ref-webcan-chart"></a>

## Chart

<br/>

The chart view lets you plot [DBC-decoded](configuration.md#ref-webcan-decode) signals in real time in customizable live charts. You can quickly change what to display via the filter panel, zoom in as needed - and seamlessly navigate the full historical buffer using the timeline slider or left/right arrow keys.

---

<a id="ref-webcan-filter"></a>

## Filter

<br/>

The filter panel lets you quickly filter your data by CAN bus, ID/PGN, message name or signals - with powerful search queries for batch filtration. Filters apply across the summary, trace and chart views above - and can be used to control what CSV data you export below.

---

<a id="ref-webcan-export-import"></a>

## Export/import

<br/>

You can export the raw / [decoded](configuration.md#ref-webcan-decode) CAN data to CSV from either the summary or trace view, with the active filters controlling the output.

The **raw trace CSV** format is particularly useful:

1. It can be re-imported into webCAN with e.g. updated DBC decoding, TP rules etc
2. It is supported via our [Python API](../python.md#ref-software-python) for e.g. replay or conversion to ASC, TRC, LOG, …
3. It matches the [CANedge](../../stream-data/canedge.md#ref-record-canedge) `mdf2csv` format (so CANedge log files can be loaded in webCAN)

---
* <a id='summarystats'>**[1]**</a> The summary statistics are calculated from the start of your session. Refreshing the webCAN page or clicking the ‘clear frames’ button will reset the statistics. However, the statistics are not bounded by e.g. the storage buffer limit. Further, disconnecting/reconnecting CAN channels will not clear the statistics

# CAN-to-MQTT

This demo forwards decoded CANsub data to an [MQTT](https://mqtt.org/) broker using `python-can` and `cantools` - e.g. for live visualization in [Grafana](https://grafana.com/).

---

## Requirements

Install the Python dependencies:

```bash
pip install python-can-cansub cantools paho-mqtt
```

You also need an MQTT broker. [Eclipse Mosquitto](https://mosquitto.org/) is a simple open-source option - start it with `mosquitto -v`. By default it only accepts local connections; to allow remote clients, add `allow_anonymous true` and `listener 1883 0.0.0.0` to the broker config.

---

## Script

The script opens a CANsub channel (or replays a log file), decodes each frame via a DBC file and publishes the signals as JSON to one MQTT topic per message (`<channel>/<message>`).

```python
import argparse
import can
import cantools
import json
import paho.mqtt.client as mqtt
import sys
import time
from pathlib import Path
from typing import Iterable
from cantools.j1939 import pgn_from_frame_id

def can2mqtt(bus: Iterable[can.Message], source_name: str, db, mqtt_host: str, mqtt_port: int, use_system_time=False):

    # Connect to the MQTT broker
    print(f"Connecting to MQTT broker: {mqtt_host}:{mqtt_port}")
    client = mqtt.Client(mqtt.CallbackAPIVersion.VERSION2)
    client.connect(host=mqtt_host, port=mqtt_port)
    client.loop_start()

    # J1939 frames embed the source address in the CAN ID, so match on the PGN
    protocol_type = db.dbc.attributes.get("ProtocolType", None)
    is_j1939 = protocol_type is not None and protocol_type.value == "J1939"
    by_pgn = {pgn_from_frame_id(m.frame_id): m for m in db.messages} if is_j1939 else {}

    topics = []
    print("\nPress \"ctrl + c\" to quit\n\nMQTT topics:")
    try:
        for msg in bus:

            # Look up the message definition (by PGN for J1939, else by CAN ID)
            try:
                if is_j1939:
                    db_msg = by_pgn[pgn_from_frame_id(msg.arbitration_id)]
                else:
                    db_msg = db.get_message_by_frame_id(msg.arbitration_id)
            except KeyError:
                continue

            # Decode the frame
            signals = db_msg.decode(msg.data, decode_choices=False, allow_truncated=True)

            # Add a timestamp in ms
            signals["timestamp"] = int((time.time() if use_system_time else msg.timestamp) * 1000)

            # Publish to MQTT (one topic per message)
            topic = f"{source_name}/{db_msg.name}"
            if topic not in topics:
                topics.append(topic)
                print(f"- {topic}")
            client.publish(topic, json.dumps(signals))
    except KeyboardInterrupt:
        pass

    client.disconnect()

def main():
    parser = argparse.ArgumentParser(formatter_class=argparse.ArgumentDefaultsHelpFormatter)
    parser.add_argument("--itf", type=str, default="cansub", help="CANsub interface, e.g. cansub:1f72d84a-usb.local@1")
    parser.add_argument("--file", type=str, default=None, help="CAN log file, e.g. log.csv")
    parser.add_argument("--dbc", type=Path, default="db.dbc", help="CAN database (DBC) file path")
    parser.add_argument("--host", type=str, default="localhost", help="MQTT broker host")
    parser.add_argument("--port", type=int, default=1883, help="MQTT broker port")
    parser.add_argument("--bitrate", type=int, default=250_000, help="CAN nominal bit-rate")
    parser.add_argument("--data-bitrate", type=int, default=1_000_000, help="CAN FD data bit-rate")
    args = parser.parse_args()

    # Load the CAN database
    if not args.dbc.is_file():
        print(f"DBC \"{args.dbc}\" not found")
        sys.exit(1)
    print(f"Loading CAN database: \"{args.dbc}\"")
    db = cantools.database.load_file(args.dbc, strict=False)

    if args.file:
        # Replay a log file (uses system time for the timestamp)
        log_file = Path(args.file)
        if not log_file.is_file():
            print(f"Cannot open log file \"{log_file}\"")
            sys.exit(1)
        reader = can.MessageSync(can.LogReader(log_file))
        can2mqtt(reader, "file", db, args.host, args.port, use_system_time=True)
    else:
        # Stream live from a CANsub channel
        interface_channel = args.itf.split(":")
        itf_name = interface_channel[0]
        itf_chn = None if len(interface_channel) < 2 else interface_channel[1]

        # Discover available channels (mDNS) and default to the first
        itf_chns = [x["channel"] for x in can.detect_available_configs(itf_name)]
        if itf_chn is None and itf_chns:
            itf_chn = itf_chns[0]
        if itf_chn not in itf_chns:
            print(f"CAN channel \"{itf_chn}\" not available")
            if itf_chns:
                print(f"Available channels: {', '.join(itf_chns)}")
            sys.exit(-1)

        print(f"Starting CANsub interface: {itf_name}:{itf_chn}")
        with can.Bus(interface=itf_name, channel=itf_chn, bitrate=args.bitrate, data_bitrate=args.data_bitrate) as bus:
            can2mqtt(bus, itf_chn, db, args.host, args.port)

if __name__ == "__main__":
    main()
```

---

## Usage

Stream live from the first detected CANsub channel:

```bash
python can2mqtt.py --dbc db.dbc --host localhost
```

Select a specific channel, or replay a log file instead:

```bash
python can2mqtt.py --itf cansub:1f72d84a-usb.local@1 --dbc db.dbc
python can2mqtt.py --file log.csv --dbc db.dbc
```

Topics can e.g. be visualized in Grafana via the MQTT datasource (URI = `tcp://localhost:1883`).
