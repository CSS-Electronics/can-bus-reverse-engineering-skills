# Introduction

The **CANsub.4** is a compact and professional CAN bus interface with four galvanically isolated CAN buses.
It combines high performance with features matching or exceeding those typically only found in larger, more expensive devices.

The device connects via either USB<sup>[1](#usb)</sup> or Ethernet, both exposing an identical network interface<sup>[2](#net)</sup>.

The device has an [LCD color display](hardware/display.md#display) providing useful information, such as CAN bus state, frame rate, and bus load.

A built-in web-based user interface (*webCAN*) can be accessed through a browser without installing any software<sup>[3](#tls)</sup>. The *webCAN* tool comes with a rich set of features, such as raw-frame view, decoded-frame view, and live plotting.

The CANsub.4 has an open [API](api/index.md#api) based on easily accessible technologies to encourage users to create 3rd-party integrations<sup>[4](#rd)</sup>.

![image](cansub4.jpg)

---
* <a id='usb'>**[1]**</a> Network over USB using the USB CDC-NCM device class.
* <a id='net'>**[2]**</a> Switching between USB and Ethernet requires only a URL change.
* <a id='tls'>**[3]**</a> A TLS certificate can optionally be installed in the browser.
* <a id='rd'>**[4]**</a> A range of 3rd party integrations and code examples are available on [csselectronics.com](https://csselectronics.com/).

# Getting started

This guide describes how to set up the CANsub.4 for the first time using a USB-C cable and the browser-based webCAN tool.

#### NOTE
Connecting via USB creates a virtual network adapter automatically - no driver installation is required. The webCAN tool and device [API](../api/index.md#api) work identically via both USB and Ethernet.

---

## Prerequisites

- Review the [usage warning](../legal/index.md#usage-warning) before connecting the device to a live CAN bus.
- Optionally, [install the CANsub root certificate](certificate/index.md#cansub-certificate-installation) to prevent browser security warnings

---

## Step 1 - power on

Connect the CANsub.4 to a PC using a USB-C cable. The display activates and shows the device network information.

![image](getting_started/cansub_display_net_usb.bmp)

## Step 2 - connect to the device

Open a browser and enter the display URL `https://[HOSTNAME].local` to open webCAN.

If the root CA certificate has not been installed, the browser will display a security warning. Accept the warning to proceed.

Connect the CANsub.4 to a CAN bus (see [connector pinout](../hardware/connector.md#can-bus)).

#### NOTE
An optional DB25-4xDB9 splitter cable is available.

webCAN includes a separate getting started guide via the ![icon_help](getting_started/help-circle.png) icon in the top right corner.

# API

The CANsub.4 has an open API based on easily accessible technologies to encourage users to create 3rd-party integrations. The API enables full control of the device.

The following diagram illustrates the CANsub stack, including the *API* layer. The built-in *webCAN* tool exclusively uses the open API.

The API is split into a [REST API](api_rest.md#rest-api) for configuration and state monitoring, and a [WebSocket API](api_ws.md#websocket-api)<sup>[1](#fn-websocket)</sup> for live data transfer (CAN bus data). The API is located under `api/`.

#### NOTE
mDNS can be used for device detection, see [Detection](../connection/index.md#detection).

---

## Usage

A typical integration follows this flow:

1. **Initialize**  *(once)*: Verify the API version and sync the device clock if needed (REST)
2. **Configure bus**  *(per bus)*: Configure CAN bus (REST)
3. **Open bus**: Connect WebSocket to open CAN bus (WebSocket)
4. **Communicate**: Communicate with the CAN bus (WebSocket) and monitor states (REST)
5. **Close bus**: Disconnect WebSocket to stop the CAN bus (WebSocket)

For more details, see [REST API](api_rest.md#rest-api) and [WebSocket API](api_ws.md#websocket-api).

---
* <a id='fn-websocket'>**[1]**</a> WebSocket is chosen over raw TCP socket to make the API web-friendly.

## Mode

The device selects which data connection to use based on the following logic<sup>[1](#fn-change-mode)</sup>:

The detection state is shown on the device display, see [Connection layout](../hardware/display.md#connection-layout).

#### NOTE
The device uses distinct hostnames for the USB (`[DEVICE-ID]-usb.local`) and Ethernet (`[DEVICE-ID]-eth.local`) connections to prevent hostname resolution conflicts when switching between the two.

---

### USB-mode

In *USB-mode* the device is powered via USB and transfers data via USB.

When transferring data over USB, the device uses the USB CDC-NCM class<sup>[3](#fn-cdc-ncm)</sup> to expose a network interface. When using this mode, the device starts a DHCP server, ready to assign an IP address to the connected USB host.

| Hostname          | DHCP   | Network       | Device IPv4   | Host IPv4    |
|-------------------|--------|---------------|---------------|--------------|
| `[DEVICE-ID]-usb` | Server | `10.x.y.0/24` | `10.x.y.1`    | `10.x.y.100` |

The device hostname and IP are shown on the device display, see [Network layout](../hardware/display.md#network-layout).

The webpage hosted by the device can be accessed via:

- `https://[DEVICE-ID]-usb.local`<sup>[2](#fn-local)</sup> (e.g. `https://aabbccdd-usb.local`)
- `https://[DEVICE-IP]` (e.g. `https://10.119.41.1`)

The CANsub.4 is identified on the USB bus by the following vendor and product IDs:

| Parameter        | Value    |
|------------------|----------|
| Vendor ID (VID)  | `0x04D8` |
| Product ID (PID) | `0xE487` |

The device ID is stored in the USB serial number descriptor.

```console
$ lsusb -v -d 04d8:e487 | grep iSerial
  iSerial                 3 aabbccdd
```

---

### Ethernet-mode

In *Ethernet-mode* the device is powered via USB (charger) and transfers data via Ethernet.

#### NOTE
An optional Power-over-Ethernet (PoE) adapter is available.

| Hostname          | DHCP   | Device IPv4   |
|-------------------|--------|---------------|
| `[DEVICE-ID]-eth` | Client | Assigned      |

The device hostname and IP are shown on the device display, see [Network layout](../hardware/display.md#network-layout).

#### NOTE
The device obtains its IP address from a DHCP server (required). The device MAC address can be found via the [label](../hardware/label.md#label) or [display](../hardware/display.md#information-layout).

The webpage hosted by the device can be accessed via:

- `https://[DEVICE-ID]-eth.local`<sup>[2](#fn-local)</sup> (e.g. `https://aabbccdd-eth.local`)
- `https://[DEVICE-IP]` (e.g. `https://192.168.1.100`)

## Detection

#### NOTE
The device ID is printed on the device enclosure and can be used to construct the device hostname as explained in [Mode](#mode) and [Getting started](../getting_started/index.md#getting-started). This section explains how to detect a CANsub device when the device ID is unknown.

The CANsub.4 advertises itself on the local network (USB or Ethernet) using mDNS<sup>[5](#fn-mdns)</sup> and DNS-SD<sup>[6](#fn-dnssd)</sup>. This allows the device to be discovered without knowing its IP address in advance.

The device registers a service of type `_cansub._tcp` with the following properties:

| Property        | Value                                                                       |
|-----------------|-----------------------------------------------------------------------------|
| Service type    | `_cansub._tcp`                                                              |
| Hostname        | `[HOSTNAME].local`<br/>(`[DEVICE-ID]-usb.local` or `[DEVICE-ID]-eth.local`) |
| Port            | `443`                                                                       |
| TXT: `api`      | REST API version (e.g. `02.00`)                                             |
| TXT: `channels` | Number of CAN-bus channels (`1`, `2`, or `4`)                               |

### Windows

On Windows, the `dns-sd`<sup>[7](#fn-dnssd-win)</sup> tool can be used for mDNS discovery.

```doscon
C:\>dns-sd -Z _cansub._tcp local

_cansub._tcp                                    PTR     aabbccdd-usb._cansub._tcp
aabbccdd-usb._cansub._tcp                       TXT     "api=02.00" "channels=4"
```

### Linux

On Linux, the `avahi-browse`<sup>[8](#fn-avahi)</sup> tool can be used for mDNS discovery.

```console
$ avahi-browse _cansub._tcp --resolve

= enx00005e005301 IPv4 aabbccdd-usb     _cansub._tcp         local
hostname = [aabbccdd-usb.local]
address = [10.201.129.1]
port = [443]
txt = ["channels=4" "api=02.00"]
```

## TLS

Data transferred over the device network connection (USB or Ethernet) is secured by TLS. The unique device certificate is signed by the root certificate below.

[`CANsub root certificate`](/../assets_collected/cansub4/cansub_root_cert.crt)
<br/>
`SHA-256 fingerprint = D064C85905195400216A11BB50D754865ED2B5F997D36A9EA2EA97883B15D0F1`
<br/>

#### NOTE
The host machine (e.g. PC) can be made to trust the device (all CANsub devices) by installing the CANsub root certificate. For more information on how to install the root certificate, see [CANsub certificate installation](../getting_started/certificate/index.md#cansub-certificate-installation).

The table below provides an overview of the root and device certificate parameters.

| Parameter           | Root certificate   | Device certificate                                       |
|---------------------|--------------------|----------------------------------------------------------|
| Key algorithm       | ECDSA P-256        | ECDSA P-256                                              |
| Signature algorithm | ECDSA-SHA256       | ECDSA-SHA256                                             |
| Issued to           | `CANsub root`      | `CANsub [DEVICE-ID]`                                     |
| Hostnames (SAN)     |                    | `[DEVICE-ID]-usb.local`<br/><br/>`[DEVICE-ID]-eth.local` |

#### NOTE
The device certificate Subject Alternative Name (SAN) field allows for hostname verification.

The device uses the TLS cipher suite: `ECDHE-ECDSA-AES128-GCM-SHA256`<sup>[4](#fn-cipher-suite)</sup>.

---
* <a id='fn-change-mode'>**[1]**</a> Changing data connection mode requires a power-cycle.
* <a id='fn-local'>**[2]**</a> `.local` is used for hostnames on a local network.
* <a id='fn-cdc-ncm'>**[3]**</a> CDC-NCM enables Ethernet networking over a USB connection.
* <a id='fn-cipher-suite'>**[4]**</a> The client must support the cipher suite for the TLS handshake to succeed.
* <a id='fn-mdns'>**[5]**</a> mDNS (Multicast DNS, RFC 6762) resolves hostnames on a local network without a DNS server.
* <a id='fn-dnssd'>**[6]**</a> DNS-SD (DNS Service Discovery, RFC 6763) advertises services on a local network.
* <a id='fn-dnssd-win'>**[7]**</a> `dns-sd.exe` is included with [Bonjour for Windows](https://support.apple.com/kb/DL999) (Apple).
* <a id='fn-avahi'>**[8]**</a> `avahi-browse` is part of the `avahi-utils` package.

# Specification

## USB

* USB-C connector
* Powered through USB connector (5 V)
* USB CDC-NCM device class

## Ethernet

* RJ45 connector
* 10/100 base-t/tx IEEE 802.3 ethernet transceiver
* Link status LED

## Display

* 0.96” backlit LCD color display

## Network

* Network over USB (CDC-NCM) or Ethernet
* Discovery via mDNS
* Data secured using TLS

## CAN bus

* Physical
  > * Four galvanically isolated<sup>[1](#fn-basic-isolation)</sup> physical CAN bus interfaces
  > * DB25 (D-sub25) connector
* Transceiver
  > * Compliant with ISO 11898-2:2016 (classical CAN and CAN-FD)
  > * Short-circuit, bus-fault, and undervoltage protection
  > * Ideal passive behavior when unpowered (high impedance / no load)
  > * TXD dominant timeout (prevents network blocking in the event of a failure)
* Controller
  > * Based on MCAN IP from Bosch
  > * Customizable bit-rate / bit-timing / sample-point
  > * Customizable hardware filters (range or mask)
  > * Support for bus-error monitoring
  > * Support for listen-only mode (transmission disabled)

## Mechanical

* Robust aluminum enclosure
* Flanges for secure mounting
* Dimensions: 67 x 87 x 20 mm (L x W x H)
* Operating temperature: -25 °C to +70 °C

---
* <a id='fn-basic-isolation'>**[1]**</a> The CANsub.4 implements a *basic* galvanic isolation, making it easy to connect CAN bus networks not subjected to high voltage differences.

## Device time

Provides access to the device time (wall-time). The device time needs to be set for the device to be able to correctly timestamp CAN bus frames.

#### NOTE
The time is reset on power-cycles and is shared across all CAN channels.

### GET /api/time

**Get device time**

Returns the current device time as an RFC3339/ISO-8601 UTC timestamp string (JSON string).

**Example request:**

```http
GET /api/time HTTP/1.1
Host: example.com
```

* **Status Codes:**
  * [200 OK](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/200) – 

    Current device time.

    **Example response:**
    ```http
    HTTP/1.1 200 OK
    Content-Type: application/json

    "2025-01-01T00:00:00.000Z"
    ```

### PUT /api/time

**Set device time**

Sets the device time using an RFC3339/ISO-8601 UTC timestamp string (JSON string).

**Example request:**

```http
PUT /api/time HTTP/1.1
```

# api_ws.html.md

# WebSocket API

The WebSocket API is used for transfer of CAN bus frames between the connected client and the CANsub.4.

#### NOTE
Device configuration and state monitoring are done via the [REST API](api_rest.md#rest-api).

---

## WebSocket usage

The `/api/can/{channel}/ws` endpoints (one for each physical channel) accept WebSocket upgrades. The WebSocket connections operate in **binary mode** only. The format of the binary data is described below.

**Connecting a WebSocket opens the associated CAN bus (channel) on the device. Disconnecting closes the channel.**

The connection is bidirectional:

- **CANsub → Client**: the CANsub sends received CAN frames, CAN transmission acknowledgements, and CAN error frames.
- **Client → CANsub**: the client sends CAN frames to be transmitted on a physical CAN bus.

### Receive

Receiving data from the WebSocket is illustrated (simplified) below. Reception follows the steps (explained in more details below):

1. Read bytes from WebSocket until a start and end boundary byte is found.
2. Perform byte de-stuffing and remove boundary bytes.
3. Validate CRC32.
4. Perform CAN frame deserialization until no more bytes.

### Transmit

Transmitting data to the WebSocket is illustrated (simplified) below. Transmission follows the steps (explained in more details below):

1. Serialize CAN frames.
2. Calculate CRC32.
3. Perform byte stuffing and add boundary bytes.
4. Write bytes to WebSocket<sup>[1](#fn-websocket-max)</sup>.

---

## Framing

Each WebSocket binary message contains one or more serialized CAN frames. Framing provides message boundary detection, payload integrity verification, and byte-stuffing.

#### Binary message structure

| Boundary byte   | Payload           | CRC32   | Boundary byte   |
|-----------------|-------------------|---------|-----------------|
| `0x7E`          | N bytes (stuffed) | 4 B     | `0x7E`          |
- `0x7E` marks the start and end of each binary message.
- *payload*: one or more serialized CAN frames concatenated (see [Serialization]()).
- *CRC32*: IEEE 802.3 CRC32 (polynomial `0xEDB88320`) computed over the raw (un-stuffed) payload bytes, appended big-endian.
- Byte-stuffing is applied to all bytes between the boundary bytes (payload + CRC32 bytes).

### Byte-stuffing

Byte-stuffing ensures that boundary byte (`0x7E`) does not appear in the message body. Before transmission, every payload and CRC32 byte is stuffed.

#### Byte-stuffing substitutions

| Original   | Encoded sequence   |
|------------|--------------------|
| `0x7E`     | `0x7D` `0x5E`      |
| `0x7D`     | `0x7D` `0x5D`      |

The escape byte `0x7D` is always followed by the original byte XOR `0x20`. On decoding, each `0x7D` causes the next byte to be XOR-ed with `0x20` to recover the original value.

---

## Serialization

CAN bus frames are serialized as specified below.

#### `CB`: Classical-Base, `CE`: Classical-Extended, `FB`: FD-Base, `FE`: FD-Extended, `ER`: Error

|     |           |        |     |     |     |     |        |    |    |    |    |    |    |   Byte |   0-5 |   6 |   7 | 8   | 9   | 10   |
|-----|-----------|--------|-----|-----|-----|-----|--------|----|----|----|----|----|----|--------|-------|-----|-----|-----|-----|------|
| Bit | 47-0      | 7      | 6   | 5   | 4   | 3   | 2      |  1 |  0 | 7  |  6 | 5  | 4  |      3 |     2 |   1 |   0 | 7-0 | 7-0 | 7-0  |

- `timestamp`: Timestamp in microseconds since 2025/01/01 00:00:00 UTC (offset: 1735689600000000 µs)
- `fdr`: FD Format Indicator
- `rtr`: Remote Transmission Request
- `brs`: Bit Rate Switch
- `esi`: Error State Indicator
- `tx`: Transmission Acknowledgement
- `dlc`: DLC
- `ide`: Identifier Extension
- `id`:  Identifier
- `err`: Error Frame
  > - `0`: Bit (bit0/bit1)
  > - `1`: Ack
  > - `2`: Form
  > - `3`: Stuff
  > - `4`: Crc

## Examples

### Single CAN frame

Encoding of a *CB* CAN frame with the following parameters:

- Timestamp: `43200123456` (2025/01/01 12:0:0.123456 in us since timestamp offset)
- ID: `1`
- Data: `0xAA`, `0xBB`

The serialized CAN frame becomes: `000A0EED9240020001AABB`.

The resulting binary message becomes: `7E000A0EED9240020001AABBA8E4A04E7E`.

### Multiple CAN frames

Encoding of two *CB* CAN frames with the following parameters:

- CAN frame 1:
  > - Timestamp: `43200123456` (2025/01/01 12:0:0.123456 in us since timestamp offset)
  > - ID: `1`
  > - Data: `0xAA`, `0xBB`
- CAN frame 2:
  > - Timestamp: `31579200123456` (2026/01/01 12:0:0.123456 in us since timestamp offset)
  > - ID: `2`
  > - Data: `0xCC`, `0xDD`

The serialized CAN frames become: `000A0EED9240020001AABB1CB89B017240020002CCDD`.

The resulting binary message becomes: `7E000A0EED9240020001AABB1CB89B017240020002CCDD77F29EFD7E`.

# connector.html.md

# Connectors

The CANsub.4 has the following physical connectors.

---

## USB and RJ45

The CANsub.4 uses a USB-C port for power. Data can be transferred using either the USB-C port or the RJ45 port (Ethernet) (see [Device connection](../connection/index.md#device-connection)).

![image](hardware/connector/cansub_power_data.png)

---

## CAN bus

#### WARNING
A galvanically isolated CAN bus must connect the isolated ground pin to the ground of the connected bus.

The CANsub.4 uses one DB25 port for the four galvanically isolated CAN buses.

![image](hardware/connector/cansub4_buses.png)

#### CANsub.4 DB25 pinout.

| Bus          | Function   |   Pin # |
|--------------|------------|---------|
| CH1          | CAN-H      |      12 |
| CAN-L        | 13         |         |
| Isolated GND | 25         |         |
| CH2          | CAN-H      |      22 |
| CAN-L        | 21         |         |
| Isolated GND | 9          |         |
| CH3          | CAN-H      |      18 |
| CAN-L        | 17         |         |
| Isolated GND | 5          |         |
| CH4          | CAN-H      |       1 |
| CAN-L        | 2          |         |
| Isolated GND | 14         |         |

#### NOTE
The DB25 connector shield is connected to the device metal enclosure. Avoid connecting it to the isolated buses.


# display.html.md

# Display

The CANsub.4 features a 0.96” color LCD display showing the current device state. The display switches between several *layouts* depending on the state of the device.

The display uses the following symbols:

| ![image](hardware/display/symbol-spinner-solid.png)    | Pending connection.                                 |
|--------------------------------------------------------|-----------------------------------------------------|
| ![image](hardware/display/symbol-usb-brands-solid.png) | USB connection active (network over USB).           |
| ![image](hardware/display/symbol-ethernet-solid.png)   | Ethernet connection active (network over Ethernet). |
| ![image](hardware/display/symbol-arrow-up-solid.png)   | Network uplink activity.                            |
| ![image](hardware/display/symbol-arrow-down-solid.png) | Network downlink activity.                          |

---

## Top bar

The top bar is persistent and shown regardless of the active layout. It provides the following information:

- Device time (format `DD/MM HH:MM:SS`).
- State of connector detection (pending/USB/Ethernet).
- Network uplink/downlink activity.

---

## Information layout

The *information* layout can be activated by pressing the button under the USB-C connector<sup>[1](#fn-info-button)</sup>. The info layout provides the following information:

- `TYP`: Device type (CANsub.4).
- `ID`: Unique device ID (8-digit hexadecimal value).
- `HW`: Hardware version.
- `FW`: Firmware version.
- `API`: API version.
- `MAC`: Unique device MAC address (same for USB and Ethernet).

| ![image](hardware/display/cansub_button.png)   | ![image](hardware/display/cansub4_display_inf.bmp)   |
|------------------------------------------------|------------------------------------------------------|

---

## Connection layout

The *connection* layout is displayed until the network connection is established (USB or Ethernet selected and IP address assigned).

`Detecting inf` (interface) remains empty until either USB or Ethernet is selected for data transfer (see [Device connection](../connection/index.md#device-connection)).

![image](hardware/display/cansub_display_itf.bmp)

`Detecting net` (network) remains empty until an IP address is obtained.

| ![image](hardware/display/cansub_display_itf_usb.bmp)   | ![image](hardware/display/cansub_display_itf_eth.bmp)   |
|---------------------------------------------------------|---------------------------------------------------------|

---

## Network layout

The *network* layout is displayed once the device network is ready and until a user has connected to the device.

| ![image](hardware/display/cansub_display_net_usb.bmp)   | ![image](hardware/display/cansub_display_net_eth.bmp)   |
|---------------------------------------------------------|---------------------------------------------------------|

#### NOTE
If connecting to the device through a browser, prefix the hostname or IP with `https://`.

E.g.:

- `https://aabbccdd-usb.local` / `https://aabbccdd-eth.local`
- `https://10.121.15.1`

The hostname depends on the active interface (USB / Ethernet).

---

## CAN bus layout

The *CAN bus* layout is displayed once a client connects to the CANsub.4. The layout shows information on all supported physical CAN buses.

![image](hardware/display/cansub4_display_can.bmp)

The example above displays the following bus information:

- **CAN1**: Nominal bit-rate: 250k, Data bit-rate 1M, 312 frames/s.
- **CAN2**: Nominal bit-rate: 500k, Data bit-rate 1M, 552 frames/s, error-passive.
- **CAN3**: Nominal bit-rate: 1M, Data bit-rate 2M, 725 frames/s, silent-mode.
- **CAN4**: Nominal bit-rate: 1M, Data bit-rate 4M, 1446 frames/s, silent-mode.

# label.html.md

# Label

A unique label is attached to each device. An example label is shown below.

![image](hardware/label/cansub_label.png)

The label contains the following information:

- `ID`: Unique device ID (8-digit hexadecimal value).
- `HW`: Hardware version (`XX.YY`).
- `DC`: Production date code (`YYYYMM`).
- `QR`: Production date code, device ID, and device MAC (`YYYYMM;[DEVICE-ID];[DEVICE-MAC]`).

#### NOTE
The QR-code can be scanned to obtain the device MAC address.
